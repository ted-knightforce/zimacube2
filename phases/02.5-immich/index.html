    Phase 2.5 — Immich: ZimaOS-to-ZimaCube 2 Migration¶

    Status: ✅ Complete
    Started: May 24, 2026
    Updated: May 24, 2026

    🎯 Goal¶

    Migrate an existing Immich photo library (14,505 photos + 925 videos, 134 GiB) from a DIY ZimaOS server to the new ZimaCube 2, preserving all albums, face tags, memories, shared links, and metadata — with zero data loss.

    📦 What Immich Stores (and What Must Move)¶

    Component Location on source Contains Must migrate?
    pgdata /DATA/AppData/immich/pgdata Full PostgreSQL database — albums, faces, people, metadata ✅ Yes — critical
    model-cache /DATA/AppData/immich/model-cache Downloaded CLIP/facial recognition models ✅ Yes — saves re-download time
    redis /DATA/AppData/immich/redis Session cache (in-memory, technically skippable) ✅ Yes — included in folder copy
    Photo library /DATA/Gallery/immich 14,505 photos + 925 videos, thumbnails, encoded video (134 GiB) ✅ Yes — the actual media
    Auto-backups /DATA/Gallery/immich/backups Daily database snapshots — last 14 kept automatically 💡 Already there — see below

    ⚠️ Critical: Moving only the photo files — without the pgdata directory — gives you all your photos but none of your albums, face tags, people, memories, or shared links. Immich treats the PostgreSQL database as the source of truth. Both must move together.

    💡 Immich automatic backups: Immich creates a database backup every day at 2:00 AM and stores the last 14 in /DATA/Gallery/immich/backups/ on your source machine. These are database-only snapshots (no photos). You may already have up to 14 days of backups sitting there right now — check before running a manual dump. A backup can also be triggered on demand via Admin → Job Queues → Create Database Backup.

    ✅ Migration Method — ZimaOS Files App (LAN Copy)¶

    This is the simplest and most reliable method when both machines are on the same LAN and running the same version of Immich from the same App Store source.

    Step 1 — Add source machine as LAN Storage on ZimaCube 2¶

    On ZimaCube 2, open the Files app: - Click + Add Storage - Select LAN Storage - Enter your source ZimaOS machine's local IP address - Authenticate and mount it

    You can now browse the source machine's files directly from ZimaCube 2's Files app.

    Step 2 — Stop Immich on the source machine¶

    Stop Immich on the source before copying. This ensures the PostgreSQL database is in a consistent, flushed state — no writes happening mid-copy.

    In ZimaOS App Store on the source → Immich → Stop.

    Step 3 — Copy the AppData and Gallery folders¶

    From the Files app on ZimaCube 2, navigate to the source machine and select:

    Source path Destination path on ZimaCube 2
    /DATA/AppData/immich /DATA/AppData/immich
    /DATA/Gallery/immich /DATA/Gallery/immich

    Select both locations, click Migrate, choose: - Overwrite at destination - Retain originals on source (do not delete source — keep it as backup until verified)

    Wait for the transfer to complete. Over 2.5GbE, ~134 GiB takes approximately 10–15 minutes.

    Step 4 — Install Immich on ZimaCube 2¶

    Open ZimaOS App Store and search for Immich.

    ⚠️ Choose the correct version. The App Store may show multiple Immich variants: - ✅ Immich (official ZimaOS/CasaOS app) — use this - ❌ BigBearCasaOS Immich — different docker-compose variables (different postgres credentials, db name, network config) — incompatible with a direct pgdata copy as my original uses Immich (the non-BigBearCasaOS version)

    The App Store version must match your source exactly. The docker-compose variables (postgres username, password, database name, network mode) are baked into the app definition — any mismatch will prevent Immich from connecting to the restored database.

    Click Install and let it complete. Start the app immediately after install — Immich will detect the existing pgdata directory and connect to it rather than initialising a new empty database.

    Step 5 — Log in with existing credentials¶

    Open Immich at http://192.168.x.x:2283.

    Log in with your existing username and password from the source instance. Your full library, albums, faces, and settings will be exactly as they were.

    Step 6 — Check Job Queues (optional but recommended)¶

    Navigate to: Profile icon (top right) → Administration → Job Queues

    The Job Queues page lists all background processing jobs. After a pgdata migration, the database already contains all previously processed results — thumbnails, face vectors, CLIP embeddings, EXIF data — so most queues will show Active: 0 / Waiting: 0.

    If any queue shows a non-zero count under Missing, click its Missing button to queue only the unprocessed assets:

    Job What it does When to run Missing
    Generate Thumbnails Large, small, blurred thumbs for each asset and person If any photos show blank previews
    Extract Metadata GPS, EXIF, resolution If any assets are missing location or date
    Smart Search CLIP ML embeddings for semantic search If smart search returns no results
    Face Detection Detect faces in each asset If face data appears incomplete
    Facial Recognition Group detected faces into people Run after Face Detection if needed
    Duplicate Detection Find similar images Optional — relies on Smart Search

    💡 If all queues show 0 — that's correct. It means your pgdata copy carried over all previously completed processing. You don't need to reprocess 15,000+ assets from scratch.

    🔧 Immich Container Configuration (ZimaCube 2)¶

    As deployed from ZimaOS App Store on ZimaCube 2:

    Container Image Tag Network
    immich-server ghcr.io/immich-app/immich-server v2.7.2 host
    immich-machine-learning ghcr.io/immich-app/immich-machine-learning v2.7.2 host
    immich-postgres ghcr.io/immich-app/postgres 14-vectorchord0.4.3 host
    immich-redis (valkey) valkey/valkey 8 host
    Container ZimaOS path Container path
    immich-server /DATA/Gallery/immich /usr/src/app/upload
    immich-machine-learning /DATA/AppData/immich/model-cache /cache
    immich-postgres /DATA/AppData/immich/pgdata /var/lib/postgresql/data

    ⚠️ Troubleshooting — Known Pitfalls¶

    Compose app fails to start — wrong image tag on app containers¶

    Symptom: immich-server or immich-machine-learning containers show tag 14-vectorchord0.4.3 instead of a valid Immich version. Compose fails immediately.

    Cause: The PostgreSQL image tag (14-vectorchord0.4.3) was incorrectly applied to the application containers. ghcr.io/immich-app/immich-server:14-vectorchord0.4.3 does not exist — Docker cannot pull it and the stack refuses to start.

    Fix: In ZimaOS App Store → Immich → Settings: - immich-server tab → Tag: change to v2.7.2 - immich-machine-learning tab → Tag: change to v2.7.2 - database tab → Tag: leave as 14-vectorchord0.4.3 ← correct for postgres

    Save → Restart.

    Photos load but albums/faces/people are missing¶

    Cause: Only the photo files (/DATA/Gallery/immich) were migrated — the database (/DATA/AppData/immich/pgdata) was not.

    Fix: Stop Immich on source. Copy /DATA/AppData/immich to ZimaCube 2. Restart Immich.

    Immich starts but database is empty (no assets)¶

    Symptom: Logging in shows an empty library. Confirmed via: 

    sudo docker exec -it immich-postgres psql -U postgres -d immich -c "SELECT COUNT(*) FROM asset;"
# Returns 0 if database is empty or freshly initialised

    Cause: Immich initialised a fresh empty database because the pgdata copy failed, landed in the wrong path, or the wrong App Store variant was installed (different postgres credentials = can't connect to the copied database).

    Fix — Option A: Re-copy pgdata ← this is the method used in this migration

    Verify /DATA/AppData/immich/pgdata exists and is populated on ZimaCube 2. Re-run the Files app Migrate from source if missing. Ensure Immich is stopped on source before re-copying.

    Fix — Option B: Official Immich pg_dump / restore ← not personally tested

    Refer directly to the official Immich documentation for the correct backup and restore procedure, including the exact commands, restore sequence, and the required search_path fix:

    👉 Immich Backup and Restore — docs.immich.app

    ⚠️ ZimaOS container name difference: The official docs use immich_postgres (underscore). ZimaOS App Store names it immich-postgres (hyphen). Substitute accordingly in any commands from the official docs.

    💡 Check for existing auto-backups first: Before running a manual dump, check /DATA/Gallery/immich/backups/ on your source machine — Immich may already have up to 14 daily backups ready to use.

    Network mode not set to host¶

    Symptom: Immich containers start but cannot communicate with each other. Server logs show database connection refused.

    Cause: Default network mode in ZimaOS App Store may be blank/bridge instead of host.

    Fix: In Immich Settings for each container tab, set Network to host. Save → Restart.

    🗄️ Verifying Database Health¶

    Connect to the Immich postgres container:

    sudo docker exec -it immich-postgres psql -U postgres

    ⚠️ The username is postgres — not postgre. A common typo produces role "postgre" does not exist and drops you back to the shell.

    Then run:

    -- Switch to immich database
\c immich

-- Refresh statistics first — n_live_tup is stale after a pgdata copy
-- and will show 0 for all tables until ANALYZE runs
ANALYZE;

-- Row counts for all tables — the primary health check
SELECT relname AS table, n_live_tup AS rows
FROM pg_stat_user_tables
ORDER BY n_live_tup DESC;

-- Verify total asset count (photos and videos are in the same table)
SELECT COUNT(*) FROM asset;

-- Verify album count
SELECT COUNT(*) FROM album;

-- Check the 5 most recently added assets
SELECT "originalFileName", "createdAt"
FROM asset
ORDER BY "createdAt" DESC
LIMIT 5;

-- Exit
\q

    ⚠️ Every statement must end with a semicolon on the same line. If you press Enter without one, psql shows immich-# and waits for more input — any subsequent line is treated as a continuation of the same statement, not a new one. Type ; and press Enter to execute, or \q to cancel and start over.

    💡 Table names are singular in the Immich schema: asset, album, person — not assets, albums, persons. Querying the plural form returns relation does not exist.

    💡 "createdAt" is when the asset was added to Immich, not when the photo was taken. The 5 most recently added assets will typically be your last import batch, not your most recently shot photos.

    📝 Option A vs Option B — When to Use Each¶

    Option A — pgdata directory copy ← used in this migration¶

    Note: This approach is not in the official Immich docs. It works because both machines ran the exact same postgres image (14-vectorchord0.4.3). The official docs only document pg_dump/restore.
    ✅ Simpler No CLI commands — done entirely through ZimaOS Files app
    ✅ Faster Direct binary copy, no SQL parsing
    ✅ Complete Preserves extensions, roles, sequences, all databases
    ⚠️ Version-locked Requires identical postgres image version on both machines
    ⚠️ Source must be stopped Copying a running postgres = corrupted data

    Option B — Official pg_dump / restore ← not personally tested¶
    ✅ Cross-version Works across different postgres versions
    ✅ Portable Plain SQL + gzip — can be stored anywhere
    ✅ Live-safe Can dump from a running database
    ✅ Official Documented and supported by Immich
    ❌ More steps Requires CLI, sed fix, specific restore sequence
    ❌ Slower Must parse and re-execute all SQL

    Use Option A when both machines run the same ZimaOS App Store Immich version (same postgres image).
    Use Option B when migrating across different Immich/postgres versions, or for regular scheduled backups.
